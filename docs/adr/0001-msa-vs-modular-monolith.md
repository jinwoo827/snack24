# ADR 0001 — MSA vs 모듈러 모놀리스

**상태:** Accepted
**날짜:** 2026-05-07
**작성자:** project owner

## 컨텍스트

snack24는 사내 간식 구독 SaaS다. 주요 도메인은:

1. **identity** — 회사·부서·직원·인증
2. **catalog** — 상품·구독 플랜
3. **order** — 주문·픽업
4. **billing** — 회사 지갑·사용 기록·청구

초기 아키텍처를 정해야 한다. 후보:

- **A. 모놀리스 (단일 Spring Boot 앱)**
- **B. 모듈러 모놀리스 (단일 앱, 도메인별 모듈로 분리, 패키지 경계 강제)**
- **C. MSA (4개 + α 서비스 분리, 자체 DB, Kafka 통신)**

## 일반적 권고

소규모 팀의 신규 프로젝트는 **B(모듈러 모놀리스)** 가 정답에 가깝다.
이유:

- 분산 트랜잭션, 서비스 디스커버리, 멀티 DB 마이그레이션 등 운영 비용 없음
- 도메인 경계가 정착되면 그때 잘라내도 늦지 않음 ("Strangler Fig" 패턴)
- 1인 팀 / 단일 배포 단위에서 MSA의 "독립 배포" 가치는 거의 0

## 본 프로젝트의 결정: **C — MSA**

다음 두 가지를 모두 인지하고 의도적으로 MSA를 선택한다.

### 선택 사유

1. **명시적 학습/포트폴리오 목적.** 지원 회사(B2B SaaS)는 실제로 MSA + Kubernetes 운영 환경을 채용 요건으로 명시. 사이드 프로젝트의 1차 목적은 "이 환경을 직접 구축·운영해본 경험" 자체다.
2. **분산 트랜잭션 학습.** Saga 패턴(Orchestration)은 모놀리스에서는 의미 없는 학습이다. order ↔ catalog ↔ billing 의 실제 분리된 트랜잭션 경계가 있어야 보상 트랜잭션·멱등성·타임아웃·DLQ 같은 실전 문제가 발생.
3. **도메인 경계 자체는 실제로 분리할 만하다.**

   | 서비스 | 변경 압력 | 데이터 민감도 | 결합도 근거 |
   |---|---|---|---|
   | identity | 낮음 (조직 구조 변경 드뭄) | 매우 높음 (PII) | 회계/주문과 다른 보안 정책 |
   | catalog | 높음 (신상품/가격 잦음) | 낮음 | 읽기 트래픽 압도, 캐시 전략 다름 |
   | order | 중간 | 중간 | 트랜잭션·동시성의 핵심 |
   | billing | 낮음 (스키마 변경 위험 큼) | 높음 (감사) | 회계 책임 분리 |

   네 도메인이 받는 변경 압력과 보안 요구사항이 실제로 다르다. 모놀리스로 두면 catalog 의 잦은 배포가 billing 의 안정성을 흔든다.

### 인정하는 트레이드오프

- **운영 비용 증가:** 6개 서비스 + 인프라 컨테이너 4개 = 1인이 다루기 부담스러운 분량.
  → 8주 일정과 K8s 학습 시간을 충분히 잡아 흡수.
- **분산 시스템의 디버깅 복잡도:** 단일 트랜잭션 `@Transactional` 한 줄로 끝날 일을 Saga 코디네이터로 풀어야 함.
  → 의도된 학습 비용. ADR 0002(예정)에서 Saga 구현 방식 결정.
- **데이터 일관성:** Strong consistency 가 아닌 eventual consistency 가 됨.
  → 대시보드 등 통계 화면은 약간의 지연을 허용. 청구처럼 중요한 흐름은 Saga 로 보장.
- **팀 규모와 어긋남:** 1인 팀 = MSA 적합도 낮음. 정직하게 README/면접에서 인정한다.

## 대안을 고른다면 어떻게 했을 것인가 (면접 대비 답변)

> "실무 신규 프로젝트라면 B(모듈러 모놀리스)로 시작했을 것입니다. 도메인 경계가 흐릿한 상태에서 MSA로 가면 잘못 잘라낸 비용이 너무 큽니다. 본 프로젝트는 학습/포트폴리오 목적이 명시적으로 우선이라 의도적으로 MSA를 골랐고, 그 점을 README와 ADR에 분명히 적었습니다."

## 부록: 서비스를 안 쪼갠 경계

다음은 의도적으로 별도 서비스로 만들지 **않은** 경계다:

- **결제 PG 연동** — billing 내부 어댑터로 흡수. 외부 시스템 연동을 별 서비스로 빼는 건 흔한 over-engineering.
- **검색** — catalog 내부 QueryDSL 으로 충분. Elasticsearch 추가는 트래픽이 정당화할 때.
- **사용자 프로필** — identity 안에 둠. 별 서비스로 분리할 가치 없음.
- **알림** — 워커(`notification-worker`)로만 분리. REST API 노출 X. 즉 "서비스"라기보단 "consumer".
